# datacodemath

Personal developer blog and portfolio. Owned web presence, fast on mobile, no CMS.

Built as part of the **Developer Visibility Stack** — a suite of tools to remove the friction between doing real work and being visible on the internet. Vibecoded with human supervision.

---

## What this is

- A portfolio + blog under a custom domain
- Publication endpoint for content generated by the companion mobile app
- Activity feed aggregating short-form platform posts (LinkedIn etc.)
- Fast static site — most pages load from CDN with zero server involvement

---

## Stack

- **Astro** — static site generator with content collections
- **Tailwind CSS** — utility styling via `@tailwindcss/vite`
- **Vercel** — hosting, CDN, and serverless function runtime
- **GitHub** — source of truth for all published content (posts are markdown files committed via API)

## Deploy to Vercel

1. Push this repo to GitHub
2. Import the repo at vercel.com — it auto-detects Astro
3. Set environment variables in Vercel project settings:

| Variable | Description |
|---|---|
| `BLOG_API_SECRET` | Random secret string — auth token for the publish API |
| `GITHUB_OWNER` | Your GitHub username |
| `GITHUB_REPO` | This repo's name |
| `GITHUB_TOKEN` | GitHub Personal Access Token with `repo` scope |
| `SITE_URL` | Your live domain, e.g. `https://datacodemath.ai` |

4. Deploy. Subsequent pushes to `main` auto-deploy.

## API

### `POST /api/publish`

Serverless endpoint. Receives content from the mobile app, writes a markdown file to the GitHub repo via the GitHub Contents API, and triggers a Vercel rebuild. Zero human intervention on the blog side.

**Auth:** `Authorization: Bearer <BLOG_API_SECRET>`

**Request body:**

```json
{
  "type": "blog_post",
  "title": "My Post Title",
  "slug": "my-post-title",
  "content_markdown": "# Hello\n\nPost body here.",
  "published_at": "2026-03-07T10:00:00.000Z",
  "tags": ["ai", "typescript"],
  "platform_variants": {
    "linkedin": "Short plain text version for LinkedIn (optional)"
  }
}
```

For `type: "activity"`, `title` and `slug` are optional — a slug is derived from the timestamp.

**Response:**

```json
{ "url": "https://datacodemath.ai/posts/my-post-title" }
```

**Errors:** `401` bad/missing auth · `400` invalid payload · `502` GitHub write failed

**CORS:** restricted to `SITE_URL`. OPTIONS preflight handled.

**Quick test with curl:**

```bash
curl -X POST http://localhost:4321/api/publish \
  -H "Authorization: Bearer your-secret" \
  -H "Content-Type: application/json" \
  -d '{
    "type": "blog_post",
    "title": "Hello World",
    "slug": "hello-world",
    "content_markdown": "My first post.",
    "published_at": "2026-03-07T12:00:00.000Z",
    "tags": ["meta"]
  }'
```

---

### `GET /api/posts.json`

Static JSON file generated at build time. Lists all published posts. No auth required — public read.

```json
{
  "posts": [
    {
      "slug": "hello-world",
      "type": "blog_post",
      "title": "Hello World",
      "publishedAt": "2026-03-07T12:00:00.000Z",
      "tags": ["meta"],
      "url": "/posts/hello-world"
    }
  ],
  "generatedAt": "2026-03-07T12:00:00.000Z"
}
```

Used by the GitHub Gardener to detect blog posts for cross-linking with repos.

## Content model

All content is markdown files in `src/content/posts/` with frontmatter:

```yaml
---
type: blog_post        # or "activity"
title: "Post Title"
publishedAt: 2026-03-07T12:00:00.000Z
tags:
  - typescript
platformVariants:
  linkedin: ""         # plain text for LinkedIn, optional
---

Post body in markdown.
```

You can write posts by hand (just add a file) or publish via the API.

## Post-deploy runbook

Run these after deploying to Vercel to confirm the live site is wired correctly.

**1. Check the posts index**
```bash
curl https://your-domain/api/posts.json
```
Expected: JSON with a `posts` array and `generatedAt` timestamp.

**2. Smoke test the publish endpoint (auth rejection)**
```bash
curl -s -o /dev/null -w "%{http_code}" -X POST https://your-domain/api/publish \
  -H "Content-Type: application/json" \
  -d '{"type":"activity","content_markdown":"test"}'
```
Expected: `401`

**3. Publish a live test post**
```bash
curl -X POST https://your-domain/api/publish \
  -H "Authorization: Bearer YOUR_BLOG_API_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "type": "blog_post",
    "title": "Deploy smoke test",
    "slug": "deploy-smoke-test",
    "content_markdown": "# Deploy smoke test\n\nIf you can read this, the publish pipeline works. Delete me.",
    "published_at": "2026-03-07T12:00:00.000Z",
    "tags": ["meta"]
  }'
```
Expected: `{"url":"https://your-domain/posts/deploy-smoke-test"}`

Wait ~30 seconds for Vercel to rebuild, then visit the URL. Delete `src/content/posts/deploy-smoke-test.md` from the repo when done.

**4. Run the full integration test suite against the live URL**
```bash
BASE_URL=https://your-domain BLOG_API_SECRET=your-secret bash tests/api/integration.sh
```
Expected: 16 passed, 0 failed.

## How this was built

This project was vibecoded with human supervision using two AI instances working in parallel, coordinated through a shared file (`convo.md`) that lives at the root of this repo.

**The setup:**

- **Claude Code** — architecture and planning. Defined the stack, API contracts, security model, content schema, and build order. Wrote briefs for Codex to implement. Did not write production code. Reviewed Codex's work for real issues.
- **Codex (Claude Code, separate instance)** — implementation. Scaffolded the Astro project, integrated the theme, built the publish endpoint, added CORS headers, wrote placeholder content. Reported back through `convo.md`.
- **Human** — made product decisions (theme choice, color direction, site name, scope), provided credentials, unblocked when choices were needed, supervised quality.

**The communication channel:**

All coordination happened through `convo.md`. Each party appended to the file — architecture decisions, implementation reports, review notes, build errors, design briefs. Neither AI instance had real-time awareness of the other; `convo.md` was the shared context both read before acting.

**Why this worked:**

Separating architecture from implementation kept each instance focused. Claude Code never touched production files (avoiding conflicts). Codex never made product decisions (avoiding drift). The human stayed out of the details unless a real choice was required.

`convo.md` is preserved as an artifact of this process. It shows the full conversation: what was decided, why, what broke, what was fixed, and in what order.
