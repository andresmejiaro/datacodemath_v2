## The Blog

**Stack:**
- **Astro** — static site generator, markdown-native, fast, ideal for developer blogs
- **Vercel** — free tier, custom domain, auto-deploys on git push, no server to manage
- **GitHub repo** — source of truth for all content (posts as `.md` files in `src/content/posts/`)

**Content delivery from App:**
Vercel serverless function at `/api/publish`:
- Accepts `POST` with `Authorization: Bearer <BLOG_API_SECRET>`
- Body (JSON): `{ type, title, slug, content_markdown, published_at, tags[], platform_posts[] }`
- Function calls GitHub API to write/create the `.md` file in the repo
- Vercel auto-detects the push and rebuilds — zero human intervention on Blog side

This satisfies every Blog constraint: git-push deploy, no paid CMS, no manual steps on receipt.

**Content types in v1:**
- `blog_post` — full markdown post, gets its own URL
- `activity` — short platform post (LinkedIn text), aggregated in an activity feed page, no dedicated URL

**Project pages:** hand-authored markdown files, not managed by App in v1.

---

## The App

**Stack:**
- **React Native** (Android v1; iOS later requires no architectural changes)
- **Backend:** Vercel serverless functions (same Vercel project as Blog, or separate — Codex decides)
- **LLM:** Claude API (Anthropic) — best fit given the use case
- **On-device storage:** SQLite via `expo-sqlite` for drafts — survives offline, survives app restarts

**Backend functions:**
- `POST /api/generate` — accepts brief + optional platform target + optional context URLs → calls Claude → returns draft(s) → App stores on device
- No backend needed for approval (pure on-device)
- `POST /api/publish` (on Blog) — called when user approves and taps publish

**Voice baseline:** On first run, user pastes 3-5 existing posts. Stored as context and prepended to generation prompts. Codex decides exact prompt structure.

**LinkedIn flow:** On approve, App uses Android Intent / Share API to open LinkedIn with pre-filled text. No LinkedIn API needed. One tap = done.

## Content Delivery Contract (the handshake)

```
POST https://<blog-domain>/api/publish
Authorization: Bearer <BLOG_API_SECRET>
Content-Type: application/json

{
  "type": "blog_post" | "activity",
  "title": "string (blog_post only)",
  "slug": "string (blog_post only, URL-safe)",
  "content_markdown": "string",
  "published_at": "ISO8601",
  "tags": ["string"],
  "platform_variants": {
    "linkedin": "plain text string (optional)"
  }
}

Response 200: { "url": "https://<blog-domain>/posts/<slug>" }
Response 4xx/5xx: { "error": "string" }
```

App retries on 5xx (up to 3x with backoff). 4xx = surface error to user, no retry.


Reads the `posts` content collection at build time. Returns all posts sorted by `publishedAt` descending.

**Response shape:**
```json
{
  "posts": [
    {
      "slug": "string",
      "type": "blog_post | activity",
      "title": "string",
      "publishedAt": "ISO8601",
      "tags": ["string"],
      "url": "/posts/slug  OR  /activity#slug"
    }
  ],
  "generatedAt": "ISO8601"
}
```

Do NOT include `content_markdown`, `platform_variants`, or any internal field. Metadata only.
